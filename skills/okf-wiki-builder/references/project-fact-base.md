# Project Fact Base Profile

Use this profile when an OKF bundle must carry durable project truth across sessions, agents, and time. These rules extend OKF v0.1; they are not official core requirements.

## Goal

Let a new reader determine quickly:

- What is known and supported.
- What was decided and why.
- What is assumed or unknown.
- Which guidance is currently in force.
- When evidence was last checked.
- Where important claims came from.
- Which concepts should be read next.

Prefer fewer, well-supported concepts over many fuzzy notes.

## Three Independent Axes

Do not collapse these into one field:

| Axis | Field | Example values | Question answered |
|---|---|---|---|
| Object kind | `type` | `Fact`, `Decision`, `Assumption`, `Runbook` | What is this? |
| Lifecycle | `status` | `active`, `deprecated`, `superseded` | Should it guide current work? |
| Epistemic state | `verification` | `confirmed`, `assumed`, `unknown` | How well is it supported? |

Use `type: Decision`, not `status: decision`. Use `status: deprecated`, not `verification: deprecated`.

## Time Semantics

Use distinct clocks:

- `timestamp`: last meaningful change to the concept content.
- `last_verified`: last time supporting evidence was checked.
- `valid_from` and `valid_until`: optional real-world validity window for time-bound facts.

An old timestamp does not necessarily mean a fact is false. A recently edited page does not prove its evidence is current.

## Recommended Fields

```yaml
---
type: Fact
title: Authentication uses session cookies
description: The application authenticates browser users with signed session cookies.
status: active
verification: confirmed
resource: repo://src/auth/session.ts
tags:
  - authentication
timestamp: 2026-07-16T10:30:00+08:00
last_verified: 2026-07-16T10:30:00+08:00
source_refs:
  - ../references/auth-design.md
---
```

Use `evidence` only for concise, machine-useful pointers. Explain important evidence and caveats in the body.

## Directory Selection

Use only directories justified by stable concepts.

Typical software project:

```text
wiki/
  index.md
  facts/
  architecture/
  modules/
  apis/
  concepts/
  entities/
  playbooks/
  runbooks/
  decisions/
  assumptions/
  references/
```

Typical data project:

```text
wiki/
  index.md
  datasets/
  tables/
  metrics/
  joins/
  dashboards/
  queries/
  decisions/
  assumptions/
  references/
```

Do not create empty categories merely to match these examples.

## Page Types

### Fact

Use for a stable statement that future work may rely on. Include Statement, Evidence, Implications, and Related Objects. Set `verification: confirmed` only after checking authoritative evidence.

### Decision

Use for a choice that constrains future work. Include Decision, Context, Alternatives Considered, Consequences, and Revisit When. Keep the rationale, not just the outcome.

When replaced, retain the old page with `status: superseded` and link the successor. The current page should link back with `supersedes` in prose or a documented local extension.

### Assumption

Use for a useful but unverified belief. Include Assumption, Why It Is Plausible, Validation Plan, and Risk If Wrong. Do not silently promote it to a fact; update its type or verification state after validation.

### Unknown

Use only for a durable, consequential gap. State why the answer matters, what was checked, and how it could be resolved. Do not turn every unanswered question into a concept.

### Runbook

Use for repeatable operational procedures. Include When To Use, Preconditions, Steps, Verification, Rollback, and Escalation when relevant.

### Reference

Use for a canonical source that supports multiple concepts or needs a durable summary. Record the source URI, scope, useful claims, limitations, and concepts that depend on it. Do not copy entire external documents.

## Evidence Rules

- Put citations close to consequential claims.
- Prefer code paths, schemas, direct observations, authoritative docs, issues, and decisions over unsourced summaries.
- Record enough detail to reproduce or recheck the claim.
- Distinguish source authority from extraction method. A claim generated by an agent is not verified merely because its source exists.
- Do not assign numeric confidence unless the bundle defines and enforces a reproducible scoring method.
- Keep contradictions visible until resolved. Do not silently choose the convenient claim.

## Retrieval And Maintenance

At the start of work:

1. Read the root `index.md`.
2. Follow the smallest relevant set of concept links.
3. Treat active confirmed facts and active decisions as constraints unless the user overrides them.
4. Verify assumptions and stale evidence before risky work.

After meaningful work:

1. Update durable facts and affected relationships.
2. Record decisions that constrain future work.
3. Mark superseded guidance instead of deleting its history.
4. Refresh indexes as derived navigation.
5. Run both OKF core and project profile lint.

## Exclusions

Do not store:

- Secrets, credentials, or private tokens.
- Stream-of-consciousness notes.
- Temporary task plans or session transcripts.
- Large copied articles or raw archives.
- Generated embeddings, search databases, or graph caches as authoritative knowledge.
- One-off command logs unless they are concise evidence for a durable fact.
