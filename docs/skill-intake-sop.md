# Skill 收录 SOP

本流程用于将外部 Skill 或插件安全、可追溯地收录进 VSIX Skills。目标用户是中国大陆地区的普通 Codex App 用户，优先保证实际价值、低使用门槛和长期可维护性。

本 SOP 规定操作顺序；具体筛选倾向见 [`skill-selection-guidelines.md`](./skill-selection-guidelines.md)，目录和字段规范见仓库根目录 [`README.md`](../README.md)。

## 基本原则

- 每批最多审核 5 个候选项，完成一批再开始下一批。
- Agent 负责初筛、溯源、许可核验、内容审核和标准化建议；用户负责最终复审和收录决定。
- 候选内容在用户确认前不得直接合入 `main`。
- 不以 Star 数、榜单排名或平台收录状态代替来源、许可和内容审核。
- 审核期间不执行上游脚本，不登录外部账号，不触发付费 API，不修改生产环境。
- Skill 和插件必须按实际产品形态收录。完整工作流已经作为插件组织时，不拆成多个互相依赖却缺少总入口的独立 Skill。

## 决策结果

每个候选项必须得到以下一种结论：

- `keep`：可以原样收录，或只需不改变核心能力的轻量调整。
- `remove`：不符合目标用户、许可、安全、依赖或维护要求，不收录。
- `needs-user-decision`：有明确价值，但存在重依赖、付费服务、地区生态、许可边界或产品形态问题，需要用户决定。

结论必须附具体原因，不能只写“合适”或“不合适”。

## 1. 建立候选批次

1. 每次选择不超过 5 个候选项。
2. 记录候选名称、发现页面、疑似上游仓库和用户提出的需求。
3. 暂不批量克隆整个平台或作者的全部仓库。
4. 同名项目较多时，先确认正确来源，再获取内容。

批次开始时应输出：

```text
批次：YYYY-MM-DD-N
候选数：1-5
候选：<id 或名称>
来源入口：<URL>
```

## 2. 溯源和版本确认

对每个候选项逐一确认：

1. 找到官方或最接近原作者的上游仓库，区分原始项目、镜像、Fork 和二次整理仓库。
2. 定位 Skill 或插件在仓库中的准确路径。
3. 记录审核所依据的 commit SHA、tag 或发布版本。
4. 判断候选项是：
   - 单一、可独立工作的 Skill；
   - 多个协同 Skill 组成的完整插件；
   - 仅供某个特定 Agent 或平台内部使用的扩展。
5. 检查上游是否存在更新版本。默认以官方稳定版本为准，不盲目采用第三方改写版。

来源信息必须保留事实准确性。不得把第三方作者改写成 VSIX，也不得为了适配 Codex 伪造不存在的上游能力。

## 3. 许可核验

1. 检查仓库根许可证、候选目录内许可证和文件头声明。
2. 确认许可证允许复制、修改和再分发当前内容。
3. 识别单独受限的图片、字体、模板、模型、数据集或第三方代码。
4. 按许可证要求保留 `LICENSE`、版权声明和必要 NOTICE。
5. 记录许可证名称和证据 URL。

以下情况不得直接复制进仓库：

- 没有明确再分发许可；
- 仓库许可证与候选子目录声明冲突；
- 核心资源受单独条款限制；
- 只能确认“代码公开”，不能确认允许再分发。

对于 Codex 官方 Marketplace 中不适合镜像的插件，可在用户批准后采用 catalog-only 外部条目，不复制受限包内容。

## 4. 内容和安全审核

阅读候选目录内全部文本文件，并清点脚本、资源、模板、示例和链接。至少检查：

- 是否包含真实 `.env`、API Key、Token、Cookie、账号或客户数据；
- 是否硬编码私人目录、内部服务器、私有网络地址或部署细节；
- 是否包含符号链接、嵌套 `.git`、`node_modules`、构建缓存或无关大文件；
- 脚本是否会上传数据、修改账号、控制浏览器、删除文件、执行远程代码或启动常驻服务；
- 文档引用的脚本、资源、命令和相对路径是否真实存在；
- 下载地址和 API 端点是否与说明一致；
- 是否属于攻击、越权、隐私侵犯、灰色自动化或其他不适合公开目录的能力。

发现疑似凭据、恶意行为或来源不明的二进制文件时立即停止审核，结论设为 `remove` 或 `needs-user-decision`，不要尝试运行。

## 5. 兼容性、依赖和地区适配

### Agent 兼容性

- 确认 Skill 能在 Codex App 或通用 Agent 环境中工作。
- Hermes 等其他 Agent 名称只出现在来源或泛化文字中时，可以做准确的中性化调整。
- 如果依赖 Hermes CLI、Hermes 配置、Hermes MCP 或其他专属运行时，不得伪造 Codex 等价物；按筛选规范移除或交由用户决定。
- 如果多个 Skill 共享路由、上下文、脚本、模板或生命周期规则，应保留为完整插件。

### 依赖分层

将依赖分成两组：

- 用户必须主动准备：系统、桌面软件、授权、服务、硬件、文件或数据资源。
- Agent 可以检测和安装：命令行工具、运行时和语言包。

重点标记 Docker、常驻服务、数据库、浏览器自动化、GPU、CUDA、模型权重、复杂 ML 环境、OAuth 应用和企业管理员权限。

### 中国大陆生态适配

- 核心结果依赖难以访问、注册或并不常用的服务，且没有合理替代方案：`remove`。
- 可以替代，但需要明显重写连接器或工作流：`needs-user-decision`。
- 仅在示例、可选集成或来源信息中出现，不影响主流程：可以保留。

## 6. 功能完整性审核

确认候选项不是只有漂亮描述而缺少可执行流程：

- 触发条件和适用任务明确；
- 执行步骤顺序清楚，Agent 知道何时停止并向用户请求输入；
- 输入、输出、失败处理和必要限制有说明；
- 脚本参数、环境变量和文档一致；
- 可选依赖不会被写成强制依赖；
- 不会把密钥要求用户公开粘贴到不合适的位置；
- 后续修改任务需要复用前一步产物时，流程能正确携带上下文。

轻量文字修正不能改变原 Skill 的真实能力。需要重写核心逻辑时，先列为 `needs-user-decision`。

## 7. 输出初审报告

完成一批候选项后，先向用户提交审核结果，不直接入库。每项使用统一格式：

```text
名称：<name>
形态：Skill / Plugin / 特定生态扩展
上游：<repository + path>
审核版本：<commit/tag/version>
许可：<license + evidence>
核心价值：<一句话>
用户前置：<无或明确列表>
Agent 依赖：<无或明确列表>
地区适配：<通过/风险及原因>
主要风险：<无或明确列表>
建议调整：<无或明确列表>
结论：keep / remove / needs-user-decision
```

用户复审后，记录最终的保留、移除或暂缓决定。未获明确批准的候选项不进入正式目录。

## 8. 标准化并入库

### 普通 Skill

1. 放入 `skills/<skill-id>/`，一个目录只包含一个可独立分发的 Skill。
2. 保留运行所需的 `SKILL.md`、脚本、资源、参考和模板。
3. 创建或规范 `skill.json`，字段以根目录 `README.md` 为准。
4. 使用简洁中文 `descriptionZh`，选择一个符合用户主要目标的分类。
5. 将用户必须处理的前置条件写入 `requirements.user`。
6. 将 Agent 可检测和安装的工具、运行时和包写入 `requirements.agent`。
7. 不添加 tags，不添加“需要支持 Skill 的 Agent”等无意义前置条件。
8. 需要环境变量时只提供 `.env.example`。
9. 上游已有有效 `agents/openai.yaml` 时保留并校验；不得填入虚假的界面信息或文件路径。

### 完整插件

1. 放入 `plugins/<plugin-id>/`，保留协同工作的内部 Skills 和共享资源。
2. 校验 `.codex-plugin/plugin.json`、内部 `skills/`、LICENSE 和引用路径。
3. 在 `.agents/plugins/marketplace.json` 中增加唯一且匹配的本地 Marketplace 条目。
4. 在 `catalog/plugins/<plugin-id>.json` 中维护网站和 Finder 使用的公开元数据。
5. 安装提示必须区分 Skill 和插件，并兼容 Marketplace 已经添加的情况，不能要求用户重复添加。
6. 官方外部插件采用 catalog-only 方式时，只记录完整 distribution 信息，不复制插件包，也不加入 VSIX Marketplace manifest。

## 9. 本地验证

提交前至少运行：

```bash
node scripts/validate-skills.mjs
node --test scripts/tests/*.test.mjs
git diff --check
git status --short
```

如果候选项自带安全、离线、无副作用的测试，也应运行。需要真实账号、付费 API、浏览器控制、云资源或生产环境的测试，不得在常规收录流程中直接执行。

人工复核 Git diff，确认：

- 只加入用户批准的候选项；
- 没有生成 ZIP、catalog、Finder API 或部署产物；
- 没有凭据、本机绝对路径、缓存和无关文件；
- 来源、许可、分类和前置条件准确；
- Skill 与插件 ID 不冲突。

## 10. 提交和发布

1. 用户确认本批结果后再提交和推送。
2. 提交信息应能看出本批新增、移除或更新的内容。
3. 推送到 `vsix_skills/main` 后，由服务器发布流程拉取、可信校验并生成 ZIP、SHA-256、catalog、Finder API 和插件 Marketplace。
4. 普通内容更新不修改 `vsix_skills_site`，也不手动生成发布文件。
5. 插件变更发布后，应在隔离的临时 `CODEX_HOME` 中验证 Marketplace 发现、安装和列表状态。

## 完成标准

一个候选项只有同时满足以下条件才算完成收录：

- 用户已明确批准；
- 来源和审核版本可追溯；
- 再分发许可明确且声明完整；
- 内容、安全、依赖和地区适配审核完成；
- Skill 或插件形态正确，没有拆散完整工作流；
- 中文描述、分类和 requirements 已规范化；
- 仓库校验和相关测试通过；
- Git diff 不包含凭据、缓存、本机路径或生成产物；
- 已提交并推送，服务端发布结果通过对应验证。
