# Storytelling, Annotation, and Critique

## What Problem This Solves

This reference helps move from "a chart exists" to "a viewer can understand what matters and why."

## When to Use It

Use this when designing explanatory figures, executive visuals, report layouts, or critique workflows.

## Key Takeaways

- Annotation should reduce interpretation cost, not merely decorate the page.
- Reports and dashboards need hierarchy: one primary claim, supporting context, and subordinate details.
- Prefer layout, labeling, and default state over prose blocks that teach the user how to read the view.
- Good critique distinguishes semantic issues from rendering or product issues.
- A visualization can be visually polished and still be analytically weak.
- A chart should survive export, resizing, grayscale, and partial reproduction.
- Editorial infographics need an insight title, an annotation plan, and a mobile reading path before rendering begins.
- Art-directed interactive stories need an artifact mode, image or illustration plan, motion purpose, static fallback, and human visual review before rendering begins.
- Sensitive geopolitical and humanitarian stories need a source and method ledger, dated evidence states, humane language, and visible separation between measured, estimated, and schematic evidence.

## Common Mistakes

- Using narrative text to compensate for a chart that does not reveal the claim.
- Using help copy or tooltip text to explain fundamentals that should be visible in the default view.
- Treating titles as labels rather than as a framing device.
- Putting equal visual weight on every tile, chart, and KPI.
- Copying a publication's surface style instead of applying its underlying principles.
- Adding generated imagery or animation because the chart feels plain, rather than because it explains scale, place, mechanism, motion, or stakes.
- Showing conflict, disaster, displacement, or civilian-harm data without dates, evidence status, caveats, or a humane static fallback.
